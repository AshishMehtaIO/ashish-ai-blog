Single-turn LLM RL is conceptually simple: prompt in, response out, reward, update. Multi-turn — where the agent generates, observes a tool or environment response, generates again, observes again, finally answers — introduces machinery the single-turn case does not need. [VERL](https://github.com/verl-project/verl) packages that machinery into one abstraction: the `AgentLoop`. This post is the working reference.

We cover the AgentLoop's coroutine state machine, the AgentData object that travels across turns, the `response_mask` trick that keeps multi-turn losses sane, tool integration via the `@function_tool` decorator and `BaseTool` lifecycle, multi-turn reward design, how async rollouts behave across turns, and the templating landmines VERL's docs explicitly warn about.<Sidenote id="sn-prereqs">Assumes you've seen single-turn RLHF/PPO before but not necessarily VERL specifically. The AgentLoop machinery lives in <code>verl/experimental/agent_loop/</code>; the "experimental" path is a stability hint we come back to in the limits section.</Sidenote>

## §1 — What changes from single-turn

In single-turn, a rollout is one autoregressive decode. The policy generates response tokens; you compute reward on the final response; you train. The trajectory is a flat sequence: prompt followed by generation.

In multi-turn, a rollout is a sequence of interactions. The agent generates a "turn" — typically text followed by a tool call or a stopping signal. If a tool was called, the framework executes it and appends the result to the conversation. The agent generates the next turn. Repeat until the agent emits a final answer or a turn limit is hit.

Three concrete things break when you naively extend single-turn machinery:

**The trajectory isn't a flat sequence.** It's interleaved generation and observation. Some tokens are the policy's actions (it should be trained on these); some are environment outputs (it should condition on these but not be trained on them).

**There can be multiple "rewards."** The terminal reward (correct answer?) is the obvious one. But tools can emit their own rewards (was this tool call useful?). Some setups want intermediate reward signals at each turn.

**Off-policy correction gets harder.** PPO's clip handles one round of staleness fine. But by turn 5 of a long rollout, the tokens generated in turn 1 were generated by a policy that was the same as the policy now training on them — but the *intermediate observations* fed into the policy decision at turn 5 depended on the world responding to turn 1's actions. If those world responses are non-deterministic, the conditional distribution drifts in ways importance sampling can't simply correct for.

VERL's design choice: provide the abstractions to handle (1) and (2) cleanly; lean on PPO's tolerance for (3) and document the caveats. The "right" theoretical answer for (3) is unsolved; VERL ships what works in practice.

## §2 — AgentLoop: the coroutine state machine

The `AgentLoop` is a class that drives one multi-turn interaction. It maintains state (the conversation so far, accumulated tokens, response masks) and steps through turns one at a time.

<AgentLoopStateMachine />

The implementation is coroutine-based. From the outside, you call `await agent_loop.run(prompt)` and get back the final state. Inside, the loop alternates between awaiting LLM generation (delegated to the rollout engine) and awaiting tool execution. The async-ness matters: hundreds of agent loops can be in flight simultaneously, and the framework schedules them cooperatively.

Files of interest:
- [`verl/experimental/agent_loop/agent_loop.py`](https://github.com/verl-project/verl/blob/main/verl/experimental/agent_loop/agent_loop.py) — `AgentLoopBase`, `AgentLoopManager`, the registration mechanism.
- [`verl/experimental/agent_loop/tool_agent_loop.py`](https://github.com/verl-project/verl/blob/main/verl/experimental/agent_loop/tool_agent_loop.py) — `ToolAgentLoop`, the concrete ReAct-style implementation.

The states the loop moves through, per turn:

1. **PROMPT**: assemble the message history (system, user prompt, prior turns).
2. **GENERATE**: ask the rollout engine for the next assistant message.
3. **PARSE**: detect tool calls in the generation (or the absence of one — the agent is done).
4. **EXECUTE**: if a tool was called, run it; collect the result.
5. **APPEND**: add the assistant message and the tool result to the conversation.
6. **CHECK**: if the agent signaled "done" or we hit the turn limit, end. Otherwise loop back to GENERATE.

After the loop ends, the assembled trajectory is what gets fed into training.

## §3 — AgentData: state that travels across turns

The state across turns is held in an `AgentData` object. Conceptually:

```python
class AgentData:
    messages: list[Message]              # the conversation, OpenAI-style
    input_ids: torch.Tensor              # tokenized so far
    response_ids: torch.Tensor           # the subset that's the policy's response
    response_mask: torch.Tensor          # 1 where the policy spoke, 0 where the env/tool did
    log_probs: torch.Tensor              # per-token log-probs the policy assigned
    turn_count: int
    images: list[Image] | None           # multi-modal additions if any
    metadata: dict
```

This is what gets handed back to the training side. The training loop sees a multi-turn trajectory as a flat token sequence — but the `response_mask` tells it which tokens to treat as actions and which to treat as observations.

The `messages` field is what the rollout engine sees on the next turn — it's the chat-formatted conversation that the model will continue from. The `input_ids` and `response_ids` are the tokenized version of the same thing, kept consistent so training doesn't need to re-tokenize.

## §4 — The response_mask trick, in depth

The single most elegant piece of design in VERL's multi-turn machinery. It is also the source of the most confusing bugs.

<ResponseMaskDeep />

Every token in the final trajectory is tagged: `1` if the policy generated it, `0` if it came from a tool, environment, or the original prompt. The loss only credits or penalizes positions where the mask is `1`. Tool outputs are part of the context the model conditioned on, but they are not part of what the policy is being trained to produce.

What this enables:

- **Multi-turn uses the same loss as single-turn.** The advantage estimator (GAE / GRPO) sees a sequence of tokens and per-token rewards. It doesn't need to know about turns. The mask handles the bookkeeping.
- **Loss aggregation is well-defined.** "Token-level loss aggregation" means summing the loss over response_mask=1 positions and dividing by the count of those positions. "Sequence-level" means averaging within each contiguous run of mask=1 tokens.
- **Importance-sampling ratios are well-defined.** The IS ratio `ρ_t = exp(new_lp_t - old_lp_t)` is meaningful only at positions the policy chose. The mask zeros out everywhere else.

What this does *not* solve:

- **Attribution within an agent turn.** If the agent's turn 2 was bad because the tool's turn 1 result was misleading, the mask doesn't tell you that — it treats every response token as equally "credited" by the response-level reward.
- **The off-policy issue from §1.** The mask makes the loss machinery clean, but the underlying statistical issue (intermediate tool responses depend on earlier policy actions) is not addressed by masking.

A subtle point worth knowing: response_mask=0 tokens still contribute to the model's *forward pass* — the policy reads them. They just don't contribute to the loss. This is why VERL has to tokenize the tool output and keep it in `input_ids` — the model needs to see those tokens to condition on them.

## §5 — Tools: the @function_tool decorator

Tools are defined as Python functions decorated with `@function_tool`. The decorator inspects the function signature and Google-style docstring, generates an OpenAI-compatible tool schema, and registers the function for use inside an AgentLoop.

```python
from verl.tools import function_tool, ToolResponse

@function_tool
async def calculator(expression: str) -> ToolResponse:
    """Evaluate a mathematical expression.

    Args:
        expression: A Python arithmetic expression, e.g. "2 + 2 * 3".
    """
    try:
        result = eval(expression)
        return ToolResponse(text=str(result))
    except Exception as e:
        return ToolResponse(text=f"ERROR: {e}")
```

That's the whole tool. The decorator:
- Inspects `expression: str` — that's an argument of type string.
- Inspects the docstring — that's the description.
- Produces the schema: `{"name": "calculator", "description": "...", "parameters": {...}}`.
- Registers the function so AgentLoop can call it.

The model sees the tool schema in its system prompt or via the OpenAI tools API (depending on how you template). When the model emits a tool call (typically a JSON object in a specific format the template enforces), the AgentLoop parses it, looks up the registered function, calls it, and feeds the `ToolResponse` back into the message history.

`ToolResponse` can carry text, images, or video — the multi-modal content gets re-injected into the next turn's prompt for the model to condition on.

### BaseTool: the lifecycle interface

For more complex tools that need setup, per-instance state, or per-step rewards, you subclass `BaseTool`:

<ToolLifecycle />

```python
from verl.tools import BaseTool, ToolResponse

class SandboxedCodeTool(BaseTool):
    async def create(self, instance_id):
        """Called once when a new agent loop starts using this tool."""
        self._sandbox[instance_id] = await create_docker_sandbox()

    async def execute(self, instance_id, parameters):
        """Called every time the agent invokes the tool."""
        code = parameters["code"]
        result = await self._sandbox[instance_id].run(code)
        return ToolResponse(text=result.stdout, ...)

    async def calc_reward(self, instance_id):
        """Optional — called after the rollout ends."""
        successes = self._success_count[instance_id]
        return 0.05 * successes

    async def release(self, instance_id):
        """Called when the agent loop ends — teardown."""
        await self._sandbox[instance_id].close()
        del self._sandbox[instance_id]
```

The four methods give you per-instance state management. The `instance_id` distinguishes between the many agent loops running concurrently — each one has its own copy of the tool's stateful resources.

`calc_reward` is the interesting optional piece. When you implement it, the tool contributes a small reward that's added to the response-level reward at the end. This is the tool-level reward signal — useful when *how* the agent used the tool matters, not just the final answer.

## §6 — Multi-turn reward design

In single-turn, rewards are simpler: one scalar per response. In multi-turn, you have choices:

**Pattern 1: Terminal reward.** Score only the final answer. The most common and the most aligned with standard RL theory. Easy to specify. The challenge: credit assignment across many turns has to be done by the algorithm, which propagates the single terminal signal back through GAE.

**Pattern 2: Per-turn reward.** Score each agent turn separately. Useful when intermediate progress is meaningful (each turn solves a subgoal). Less common; requires either a step-level verifier or a process reward model.

**Pattern 3: Hybrid.** Most production setups: a terminal task reward, plus small per-turn rewards from tools (via `calc_reward`), plus per-step format checks. All additive, all controlled by their own weights.

Two design pitfalls specific to multi-turn:

- **Tool-use overweighting.** If `calc_reward` gives a small bonus per tool call, the policy will learn to call tools gratuitously. Mitigation: bound `calc_reward` to be small relative to task reward; penalize unnecessary tool calls.

- **Length explosion across turns.** Multi-turn responses are *much* longer than single-turn. Format penalties tuned for single-turn no longer make sense; KL coefficients that worked for short responses are too weak for long ones. Re-tune for the multi-turn distribution.

## §7 — Async rollouts in multi-turn

The intro post in this series introduced async rollouts. In multi-turn, async-ness is more nuanced.

**Within a single rollout**, the AgentLoop is already async — generation and tool execution overlap (the tool runs while the GPU is idle waiting for it; the GPU continues other agent loops while one is in tool-wait).

**Across rollouts**, the same async rollout patterns apply as in single-turn: one-step-off (the next batch's generation overlaps the current batch's training) and fully async (continuous generation, opportunistic training).

What multi-turn adds: rollouts have *variable length*. A simple math problem might take 2 turns; a hard reasoning problem might take 8. This means batches finish at different times. VERL handles this with two mechanisms:

- **Dynamic batching at the rollout level**: as agent loops finish, they're removed from the active set and new ones are added. The rollout engine doesn't wait for the slowest loop in a batch.
- **Turn limits**: a hard maximum on turns prevents pathological loops from blocking training indefinitely.

Async multi-turn is where the "one-step-off async" abstraction starts to get strained — staleness now depends on how long a rollout takes, not just on how recently weights were synced. The fully-async mode handles this more gracefully but at the cost of more variable staleness. We cover the details in [the scaling post](/ashish-ai-blog/blog/verl-scaling-async/).

## §8 — Templating landmines

The templating layer — how the conversation is formatted into the prompt the model sees — is the source of more bugs in multi-turn than any other layer. VERL's docs are explicit about this.

**Landmine 1: delta tokenization.** Each turn's tokens are appended to the accumulated `input_ids`. But the tokenizer doesn't always produce the same tokens for the same string when there's preceding context — special tokens, BOS handling, leading whitespace can all cause tokenization to differ from what you'd expect. If your post-processing assumes "the tokens generated this turn = the tokens generated when re-tokenizing this turn's string alone", you can hit subtle off-by-one errors that corrupt the response_mask.

Mitigation: trust `response_ids` from the rollout engine; don't re-tokenize.

**Landmine 2: chat template vs raw template.** Some models use a chat template (Llama-style `<|begin_of_text|><|start_header_id|>user<|end_header_id|>`). Others use raw text with role markers. The template you train with *must* match the template you deploy with — production inference using a different template gives the trained policy out-of-distribution prompts.

Mitigation: pin the tokenizer's `chat_template` in your config. Test rollouts at training time match production at inference time before launching long runs.

**Landmine 3: tool-call format mismatch.** The model is prompted to emit tool calls in a specific format (JSON, XML, function-call syntax). If the rollout engine parses one format and production inference expects another, you have a model that's trained to produce a format nothing downstream knows how to handle.

Mitigation: use the same `tools` parameter end-to-end. If you use vLLM's `tools` API at training time, use it at production time too.

**Landmine 4: reasoning content vs response content.** Recent models distinguish "thinking" tokens from "final response" tokens (DeepSeek-R1 style). If your training treats them all as response, but production strips thinking, you're training a different distribution than you're serving.

Mitigation: align reasoning-content handling between train and prod. Document the template carefully. The VERL docs flag this as a known footgun.

## §9 — Conversation branches and the tree view

A useful mental model for multi-turn rollouts is as a tree. The prompt is the root. Each generation node has children for the possible tool responses (in deterministic settings, one child per tool call; in stochastic settings, multiple).

<ConversationTree />

In VERL, we typically don't explicitly explore branches at rollout time — each rollout walks one path through the tree. But the tree view helps explain a few things:

- **Why GRPO group sampling matters more in multi-turn**: each rollout takes one path, so the variance across rollouts of the same prompt is higher than in single-turn. Group sampling averages over more diverse paths.

- **Why tool-use diversity matters**: if every rollout takes the same tool path, you're training one path. Diverse temperature, top-p sampling, and varied tool affordances help cover the tree.

- **Future direction: tree-search rollouts**: some research (MCTS-style rollouts during RL training) explicitly branches the tree. VERL doesn't ship this yet but the abstraction is compatible.

## §10 — Limits and current state

What multi-turn does well in VERL today:
- The `response_mask` trick is clean and the abstractions hold.
- Tool integration is genuinely simple (just decorate a function).
- Async rollouts work and yield real throughput gains.

What's experimental / rough:
- **The AgentLoop is in `experimental/`.** The API can change. Code against it expecting maintenance.
- **No env abstraction.** Tools play the role of environments, but there's no native `Env.reset()` / episode-boundary semantics. If you need that, you wrap it inside a tool that remembers its own state.
- **Templating is do-it-yourself.** The landmines above are real. Production deployments need careful diff testing between training and serving templates.
- **Multi-turn reward design is unstandardized.** Best practices are emerging; no one-size-fits-all recipe.

## §11 — A mental model

Multi-turn in VERL is *the same training loop with two adjustments*: the rollout produces interleaved generation+observation; the loss machinery is told (via `response_mask`) which positions to credit. Everything else — algorithms, rewards, backends, the rest of the framework — is unchanged.

The AgentLoop is the right abstraction. Tools-as-functions is the right interface. The response_mask is the right trick. Where the design strains is in the templating layer (because templates are inherently model-specific) and in async semantics across long rollouts (because staleness across turns is an unsolved problem).

If you're building a multi-turn agent on VERL, the practical advice is: start with `ToolAgentLoop` and the function-tool decorator; use the GSM8K calculator example as your reference; train with GRPO and a small terminal reward plus tool-level rewards; watch the response_mask coverage in your logs; pin your chat template; and assume you'll discover one templating bug per week for the first month.
